# GEMINI.md — ArcInvestments Developer & AI Guide

This document defines coding standards, command references, architecture patterns, and design systems for the **ArcInvestments** project, customized for development with Gemini.

---

## Project Overview

**ArcInvestments** is a dark-themed, frontend-only React + Vite stock market dashboard for three Thai indices: **SET100**, **sSET**, and **MAI** (~254 stocks total).
- **Core Aesthetic**: Professional financial terminal with compact grids, strict border systems, and minimal padding.
- **Data Flow**: Core logic reads from the `src/data/` layer. The frontend is decouple-ready for live API integration.
- **Data Source**: Combined real Yahoo Finance data (fetched via python scripts) and a pseudo-random seeded mock fallback.

---

## Development Environment & Commands

Always use the project management scripts `ArcInvestments.ps1` (PowerShell) or `ArcInvestments.sh` (Bash) to run and check servers.

### 1. Dev Server (Vite)
Runs on port **5179** (strict port).
* **Start**:
  ```powershell
  .\ArcInvestments.ps1 dev start
  ```
  or
  ```bash
  ./ArcInvestments.sh dev start
  ```
* **Stop**:
  ```powershell
  .\ArcInvestments.ps1 dev stop
  ```
* **Status**:
  ```powershell
  .\ArcInvestments.ps1 dev status
  ```

### 2. Production Preview Server
Runs on port **4179**. Builds automatically if `dist/` is missing.
* **Start**:
  ```powershell
  .\ArcInvestments.ps1 prod start
  ```
* **Stop**:
  ```powershell
  .\ArcInvestments.ps1 prod stop
  ```

### 3. Data Fetching Pipeline
Fetches fresh price history and financial statements from Yahoo Finance, caching them in `src/data/real/`.
* **Fetch All Indices**:
  ```bash
  python fetch_data.py
  ```
* **Fetch Specific Index**:
  ```bash
  python fetch_data.py --index set100
  python fetch_data.py --index sset
  python fetch_data.py --index mai
  ```
* *Constraint*: Never edit files under `src/data/real/` manually; they are regenerated by this pipeline.

## Design System (Financial Terminal Theme)

All elements must adhere to the high-density financial terminal look. No rounded-xl corners or large soft shadows. Use sharp gridlines.

| Variable Name | Utility Hex | Purpose |
|---|---|---|
| Background | `#0A0E17` | Default page background |
| Surface | `#111827` | Card, container, and dialog backgrounds |
| Border | `#1F2937` | Divider grids, boundaries, and table borders |
| Accent | `#3B82F6` | Selection indicators, highlights, and primary CTAs |
| Bullish | `#10B981` | Upward movements, positive values, buy signals |
| Bearish | `#EF4444` | Downward movements, negative values, sell signals |
| Muted | `#6B7280` | Non-critical labels and descriptions |
| Body Text | `#D1D5DB` | Primary details, text descriptions |
| Heading Text | `#F9FAFB` | Major labels, stock names, section titles |

- **Standard UI Text**: `Inter` (Sans-serif)
- **Financial/Numeric Data**: `JetBrains Mono` (Monospace, alignment critical)

## Coding Rules & Implementation Standards

### 1. React & Component Conventions
* **Functional Components Only**: Standard React functional declarations with arrow syntax or `function` keyword. No class components.
* **Exports**: Always use named exports. The only exception is `App.jsx` which utilizes default export.
* **Component Length**: Maintain components under 300 lines. If a file grows larger, decompose it into helper components.
* **Inline CSS**: Never use inline `style` objects except when explicitly required for dynamic layouts (e.g. dynamic dimensions calculated in JS). Use Tailwind utilities.

### 2. State & Data Layer Guidelines
* **Zustand Store**: `useAppStore.js` is the single source of truth. Keep filters, selections, and screeners in the store. Do not sync local state for fields that impact table filtering.
* **Separation of Concerns**: Never hardcode formulas or mock data objects directly in component structures. Always place calculations in `src/data/` or hooks.
* **Determinism**: Never use `Math.random()` to generate stock details. If mock data is needed, use the `mulberry32` seeded PRNG in `mockPriceHistory.js`.

### 3. Math & Indicator Standards (`indicators.js`)
* Indicator functions must be **pure functions** (same input array yields the same output array).
* Maintain JSDoc formatting specifying parameters and return structures.
* Return arrays must match the length of the input array. Use `null` padding for warm-up periods:
  * RSI (14) → first 13 elements must be `null`.
  * EMA/SMA (Period P) → first `P - 1` elements must be `null`.

### 4. Lightweight Charts (TradingView) Integration
* Always cleanup chart objects during React unmount phases to prevent memory leaks:
  ```js
  useEffect(() => {
    const chart = createChart(containerRef.current, chartOptions);
    // ... setup series ...
    return () => {
      chart.remove();
    };
  }, [...]);
  ```
* Toggleable indicators should be dynamically added/removed or updated via the stored series reference instead of fully recreating the chart canvas.

### 5. Performance Targets
* **Load Speeds**: Ensure initial load stays under 2 seconds.
* **Memoization**: Cache heavy filtering algorithms in `useFilteredStocks.js` and indicator computations in `useStockData.js` using `useMemo`.
* **Terminal Launch**: The `<StockTerminal>` modal must load under 500ms when user clicks a row. Lazy-load it via `React.lazy` and `Suspense`.

## Verification Checklist

When editing the application, always test:
1. **Indicator Rendering**: Check that EMA, SMA, RSI, and MACD render properly without offsetting candles.
2. **Horizon Consistency**: Changing the Trend Horizon in the TopBar must update the Trend column values appropriately based on `ipoPrice`, `5y`, `1y`, or `ytd` reference points.
3. **Data Completeness**: If the mock data hiding toggle (`hideMockData` in Zustand) is active, confirm that only stocks with fetched JSON outputs are displayed.
4. **Keyboard Accessibility**: Ensure rows can be navigated using `Up/Down` arrow keys and opened with `Enter`. Esc should close the Stock Terminal modal.
